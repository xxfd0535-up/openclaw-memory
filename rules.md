# rules.md - 运维原则

## 核心原则

### 1. 先建认知，再行动

```
禁止：
- 不清楚当前状态就执行操作
- 跳过诊断直接猜测修复

必须：
- 先读取日志/配置
- 确认运行环境状态
- 建立完整认知后再行动
```

### 2. 最小变更原则

```
禁止：
- 大规模重构
- 同时修改多个不相关文件
- 不可回滚的修改

必须：
- 一次只改一个关键点
- 改完立即验证
- 能回滚到上一状态
```

### 3. 诊断优先原则

```
所有问题处理顺序：
1. 观察（不臆测）
2. 日志分析（找证据）
3. 定位（确定环节）
4. 方案（最小化）
5. 执行
6. 验证
7. 记录
```

### 4. 配置只读原则

```
配置文件（如 openclaw.json）：
- 只读，不直接修改
- 如需修改，先申请用户授权
- 修改前先备份
```

### 5. 版本化原则

```
所有关键变更必须：
- 先 commit 原版
- 再执行修改
- 能追溯每一版历史
```

### 6. 故障积累原则

```
每次故障修复后：
- 记录问题现象
- 分析根因
- 记录修复动作
- 总结教训
- 更新 incident.md
```

## 命令执行规范

### WSL 命令格式

```bash
wsl -d Ubuntu-E -u administrator -- bash -lc "<command>"
```

### 优先检查顺序

1. 进程状态：`ps aux | grep openclaw`
2. 端口监听：`ss -tlnp | grep openclaw`
3. 日志文件：`/tmp/openclaw/openclaw-YYYY-MM-DD.log`
4. 配置完整性：`cat /home/administrator/.openclaw/openclaw.json`

### 禁止行为

- 禁止 `pkill -f openclaw` 暴力杀进程（可能丢数据）
- 禁止直接编辑运行中配置文件
- 禁止跳过日志分析直接猜测原因
